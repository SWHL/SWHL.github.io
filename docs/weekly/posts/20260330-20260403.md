## Claude Code 源码泄露

> 在发布 npm 包 `@anthropic-ai/claude-code v2.1.88` 时，误将一个约 60 MB 的 .js.map（Source Map）调试文件打包进生产版本。
>
> 该 Source Map 文件本应在构建后被排除，但由于 .npmignore 配置缺失，导致包含 1900 多个 TypeScript 源文件、总计 51.2 万行代码 的完整项目结构被公开暴露。

Claude Code 主要开发者鲍里斯切尔尼 (Boris Cherny) 的一句话让我很有感触：

> 与其他任何事件一样，解决问题的反直觉方法是寻找更快的方法，而不是引入更多流程，在这种情况下就是要提高自动化程度，并且由 Claude Code 检查结果。

我已经将我维护的开源项目单元测试和发包步骤全部由 Github Actions 来实现了。重复的工作尽量减少手动操作。因为人类很容易在重复事情中犯错。人类也是流程中最容易带来不确定性的变量。

还有一个有趣的事情是：最近开源的项目，大多都是围绕 Claude Code 展开的。可是 Claude Code 背后调用的是闭源模型。这一点就像是一根线上飞着很多风筝。一旦这根线有所变动，这些风筝将直接受到波及，没有任何缓冲余地。同时，这些项目在过一段时间，就会淹没在开源的海洋里，没有人会记得。在 https://www.claudescode.dev/ 网站中，统计汇总了 Claude Code 代码生成工具全球开发活动的公开数据。从这些数据中，可以看到贡献的大部分仓库都是 Star < 2 的仓库。

## AI 工具发展观察及思考

在去年 11 月之前，AI 编码大模型远没有现在这么火爆，只因为模型能力还没有跨过某个线。显然，在 11 月后，模型能力跨过了这个临界点，一切变得不一样起来。从用户角度来看，让 AI 写的代码，基本都是对的了，甚至比自己写得还要好。

那么，当大家人手一个很强的工具后，真正决定差距和上限的东西，便开始从模型能力重新回到使用者本人身上。这就像一把锤子，当它顺手到不需要思考，你不会再研究锤子，而是开始关注钉子钉得怎么样了。从我自身感受来说，我手里有一个利器，脑海里很多想法开始萌动，但是哪些事情是真的值得去做、值得花很多年去换，反而变得更难想清楚。

个人比较赞同 George Hotz 的观点：

> Anything a person without skill can build with AI is worth very little, because anyone else can build that same thing. However, people with skill can use the same tools and build valuable things, many top photographers in the world today use iPhones.
>
> 没有技能的人用 AI 构建出的东西价值也很小。因为任何人都可以构建同样的东西。但是，有技能的人使用同样的工具却能创造有价值的东西来。世界上很多顶级摄影师都用的是 iphone。

另外一个角度是“活人感”，或者说是是否真的用心，来看现有的作品。很多作品，你一眼看到就知道这是 AI 做的。或许在以后，手工反而是稀缺的那个。

这个最直接的例子，便是 F1 车手 Lando Norris 的个人官网（http://landonorris.com/）。这个网站是一个被精心打磨到极致的作品。你一眼就知道，这个不是模板做的。
